# 調査レポート: BIGLOBE IPv6オプションとWXR9300BE6Pの状態分析

**作成日**: 2026-01-09
**目的**: WXRのMAP-E接続失敗の原因調査

---

## 1. WXR9300BE6Pの現在の状態

### ステータス画面から読み取れる情報

| 項目 | 値 | 分析 |
|------|-----|------|
| 動作モード | ルーターモード | 正しい |
| IPアドレス取得方法 | インターネット@スタートを行う | 自動判別モード |
| 接続方法 | インターネット接続可能なDHCPサーバーもしくはPPPoEサーバー未検出のため判別不可 | **問題点** |
| 有線リンク | 10GBASE-T (全二重) | 正常 |
| LAN側IP | 192.168.100.1 | 正しい |
| DHCPサーバー | 無効 | 正しい |
| **IPv6接続方法** | **IPv6ネイティブ接続** | **v6プラス/IPv6オプションではない** |
| IPv6接続状態 | 通信中 | IPv6自体は取得できている |
| グローバルアドレス | 2404:7a82:4d02:41f0:f2f8:4aff:fe67:5800/64 | 取得済み |

### 重要な発見

**WXRは「IPv6ネイティブ接続」になっている。これはMAP-E (v6プラス/IPv6オプション) ではない。**

- IPv6ネイティブ = 純粋なIPv6接続のみ (IPv4 over IPv6なし)
- MAP-E (v6プラス/IPv6オプション) = IPv6上でIPv4を通す技術

---

## 2. BIGLOBE IPv6サービスの種類

| サービス名 | 技術 | IPv4処理 | 対応機器 |
|-----------|------|----------|----------|
| **IPv6オプション** | MAP-E | IPv4 over IPv6 (IPoE) | 必要 |
| **IPv6オプションライト** | IPoE + PPPoE | IPv4はPPPoE | 不要 |
| v6プラス | MAP-E | IPv4 over IPv6 (IPoE) | 必要 |

### v6プラスとIPv6オプションの違い

- **v6プラス**: JPIX社の設備を使用 (新規受付終了)
- **IPv6オプション**: BIGLOBE社の設備を使用 (現行サービス)
- **技術的には同じ**: どちらもMAP-E方式

**参考**: [ビッグローブ光のIPv6オプションとv6プラスの違い](https://big-navi.com/biglobe-ipv6)

---

## 3. なぜWXRが「IPv6ネイティブ接続」になったのか

### 考えられる原因

1. **MAP-Eの認証競合**
   - VyOSが先にDHCPv6-PDでプレフィックスを取得
   - BIGLOBE側でMAP-E認証がVyOSに紐づいた
   - WXRはMAP-Eの認証情報を取得できず、フォールバックでIPv6ネイティブに

2. **回線判別の失敗**
   - WXRの「インターネット@スタート」がv6プラス/IPv6オプションを検出できなかった
   - IPv6は取得できたが、MAP-Eトンネルは確立できなかった

3. **プロバイダ側の制限**
   - 1回線につき1台のMAP-E対応ルーターのみサポート
   - 複数ルーターがMAP-Eを要求すると、後発は失敗する可能性

### Buffalo公式の見解

> "MAP-E機能を搭載している市販の機器においてもIPv6オプションをご利用いただけますが、すべての機種でIPv6オプションを使えることを保証するものではありません"

**参考**: [IPv6オプション対応ルーター](https://support.biglobe.ne.jp/ipv6/compatibility.html)

---

## 4. IPv6ネイティブ接続の意味

### WXRステータスの「IPv6ネイティブ接続」とは

Buffalo設定画面の選択肢:
- IPv6を使用しない
- インターネット@スタートを行う / IPv6ブリッジを許可する
- NTTフレッツ光ネクストを使用する
- **IPv6ネイティブを使用する** ← 現在これになっている
- NDプロキシを使用する
- IPv6ブリッジを使用する

**IPv6ネイティブ** = DHCPv6/SLAACでIPv6アドレスを取得するが、MAP-Eトンネルは張らない

### 結果

- IPv6インターネット: 通信可能 (2404:7a82:4d02:41f0::/64)
- IPv4インターネット: **通信不可** (MAP-Eトンネルがないため)

---

## 5. 現状の問題整理

```
現在の状態:

[NGN] ─── [LXW-10G5] ─┬─ [VyOS eth1]
                      │   ├ DHCPv6-PD: 2404:7a82:4d02:41xx::/56 取得
                      │   ├ MAP-E: (不明、要確認)
                      │   └ IPv6: 正常
                      │
                      └─ [WXR 10G]
                          ├ IPv6: 2404:7a82:4d02:41f0::/64 取得
                          ├ MAP-E: 失敗 (IPv6ネイティブにフォールバック)
                          └ IPv4: 不通
```

**VyOSのIPv6が動いているのは、VyOSがMAP-Eを使っていないから。**

VyOSはIPv6ネイティブ + DHCPv6-PDでプレフィックスを取得しているだけで、MAP-Eは使っていない。

---

## 6. 解決策の選択肢

### 選択肢A: VyOSを一時停止してWXRにMAP-Eを取らせる

1. VyOSのDHCPv6を無効化
2. WXRを再起動して回線判別をやり直す
3. WXRがMAP-E (IPv6オプション) を取得できるか確認

**リスク**: VyOSのIPv6が止まる

### 選択肢B: WXRをブリッジモードにしてWi-Fi APのみにする

1. WXRをAPモードに変更
2. VyOSでMAP-Eを実装
3. IPv4もVyOS経由

**リスク**: VyOSでのMAP-E実装が複雑

### 選択肢C: IPv6オプションライトに変更

1. BIGLOBEに連絡してIPv6オプションライトに変更
2. IPv4はPPPoE、IPv6はIPoEの併用
3. VyOSでPPPoEクライアント設定

**メリット**: MAP-E競合問題を回避
**デメリット**: IPv4がPPPoE (混雑時に遅くなる可能性)

### 選択肢D: 1台構成にする

- VyOSのみ、またはWXRのみでネットワークを構成
- 競合問題を根本的に解消

---

## 7. 結論と推奨

### 現状の診断

WXRが「IPv6ネイティブ接続」になっているのは、**MAP-E (IPv6オプション) の認証/検出に失敗した**ため。原因はVyOSとの競合、またはプロバイダ側の制限の可能性が高い。

### 次のステップ

1. **まずVyOSのDHCPv6を一時無効化してWXRの回線判別を再実行**
   - これでWXRがMAP-E取得できれば、競合が原因と確定

2. **それでもダメなら、BIGLOBEに問い合わせ**
   - 複数ルーターでのIPv6オプション利用可否を確認

3. **最終手段: 構成の見直し**
   - 選択肢B (VyOSでMAP-E) または選択肢C (IPv6オプションライト)
